# M012: Admin Polish + Mobile-First Sites

**Gathered:** 2026-03-17
**Status:** Ready for planning

## Project Description

BuilderMonster admin panel for managing TSA Amazon affiliate site portfolios. Generates fully static Astro sites served from Hetzner VPS. This milestone addresses gaps between the existing DB schema (which is already correct) and the admin forms that expose it, plus makes generated sites genuinely mobile-first, and seeds legal templates with dynamic placeholder substitution at build time.

## Why This Milestone

Several capabilities exist in the DB but are unreachable from the admin UI:
- Product content fields (`detailed_description`, `pros_cons`, `user_opinions_summary`, `meta_description`) are generated by ContentGenerator but not editable
- Category `meta_description` (stored in `description` per D057) has no form field
- Homepage SEO text has no column and no editor
- Legal templates render as plain text despite the editor hinting markdown support
- Settings page is a single long scroll with no organisation
- Agent system prompts show empty when no DB override exists — the user can't see the active default
- All three site templates have nav that breaks on mobile and CTAs that aren't full-width

## User-Visible Outcome

### When this milestone is complete, the user can:

- Edit all AI-generated content fields on a product (description, pros, cons, user opinions, meta) directly in the form, with Generate with AI populating the editable fields
- Add a meta description to a category from the category edit form
- Edit the homepage focus keyword and SEO body text from the site edit page
- Open Settings and immediately see which tab contains what — API Keys, AI Prompts, Deployment
- See the active system prompt (default or override) in the AI Prompts tab and edit it directly
- Create and edit legal templates with a live markdown preview and know which placeholders are available
- Deploy a site and have the legal pages rendered as formatted HTML with the site name, domain, and contact email substituted automatically
- View any generated site on a 375px mobile viewport with a working hamburger menu, full-width CTAs, and no broken layouts
- Use `tsa/classic`, `tsa/modern`, `tsa/minimal` as template slugs — namespacing is consistent for future site types

## Completion Class

- Contract complete means: forms save the right DB columns; `homepage_seo_text` migration applied; template slugs migrated; legal seed inserted; markdown renders as HTML in `[legal].astro`; hamburger JS toggles correctly
- Integration complete means: Generate with AI populates editable product fields; placeholder substitution works in legal pages on a full site build
- Operational complete means: none beyond standard deploy

## Final Integrated Acceptance

- Edit product fields → save → regenerate site → product page shows updated description, pros/cons
- Legal pages on generated site show `{{site.name}}` replaced with actual site name, markdown headings and lists rendered as HTML
- Open admin on a 375px viewport → hamburger opens/closes nav correctly on all three templates

## Risks and Unknowns

- `pros_cons` is JSONB in DB — ProductForm must serialize two textareas (pros lines, cons lines) into `{pros: string[], cons: string[]}` on save and deserialize back on load — easy but must not regress existing AI-generated pros/cons
- Markdown → HTML in `[legal].astro` requires `marked` in the generator — trivial install, zero runtime risk (static build)
- Template slug migration from `classic` → `tsa/classic` must update both `site_templates` table and existing `sites.template_slug` values atomically — needs a single migration with UPDATE + INSERT
- Mobile hamburger requires inline `<script>` in each layout or a shared JS snippet — must not break Astro's static output or add a build-time dependency
- `company_name` and `contact_email` are in `sites` table but not in `SiteInfo` type in `data.ts` — must be added to the type and to the `site.json` assembly in `generate-site.ts`

## Scope

### In Scope

- DB migrations: `homepage_seo_text` on `sites`; `meta_description` on `tsa_products`; template slug rename to `tsa/*`
- ProductForm: editable content fields + Generate with AI populating them (not read-only preview)
- CategoryForm: `meta_description` field (saves to `description` column per D057)
- Homepage SEO section in `/sites/[id]/edit` with `focus_keyword` + `homepage_seo_text` + Generate with AI
- Settings tabs: *API Keys* | *AI Prompts* | *Deployment* (Cloudflare + Hetzner)
- Agent system prompts: show hardcoded default as `defaultValue` so user sees what's active; save override to DB; delete row on clear to restore default
- Legal template editor: markdown preview toggle + placeholder hint panel listing all available `{{placeholders}}`
- Legal template seed migration: 8 templates (4 types × ES + EN) with `{{site.name}}`, `{{site.domain}}`, `{{site.contact_email}}`, `{{site.affiliate_tag}}` placeholders
- `interpolateLegal(content, site)` helper that substitutes placeholders before markdown→HTML conversion
- `marked` added to `apps/generator` for HTML conversion at build time
- Mobile-first: hamburger nav with inline JS (all 3 templates), full-width CTAs on product pages, pros/cons `grid-cols-1 sm:grid-cols-2` everywhere
- `company_name` + `contact_email` added to `SiteInfo` type and `site.json` assembly in `generate-site.ts`
- `generate-seo-text` API route: new `field: 'homepage_seo_text'` case

### Out of Scope / Non-Goals

- WYSIWYG editor for legal templates
- New site types beyond TSA
- Image gallery in ProductForm
- Per-template mobile variants (one responsive approach per template is sufficient)

## Technical Constraints

- `marked` in `apps/generator` only; admin uses dynamic `import('marked')` on client for live preview (no CDN dependency)
- Hamburger JS must be `<script is:inline>` — no separate bundle
- All migrations additive; no destructive column changes
- Template slug migration: single SQL file with INSERT of new `tsa/*` slugs + UPDATE of existing `sites.template_slug` rows + optionally DELETE old bare slugs if no FK references remain

## Integration Points

- `generate-site.ts` (`@monster/agents`) — SELECT must include `company_name`, `contact_email`, `homepage_seo_text`; `SiteInfo` shape in `site.json` gains these fields
- `[legal].astro` — switches from `{pageContent}` to `set:html={marked(interpolateLegal(pageContent, site))}`
- `generate-seo-text` API route — new `homepage_seo_text` field case fetches `sites` row and generates homepage copy

## Available Placeholders in Legal Templates

```
{{site.name}}          — Site name (e.g. "Mi Tienda de Freidoras")
{{site.domain}}        — Domain (e.g. "freidoras.com")
{{site.contact_email}} — Contact email from sites.contact_email
{{site.affiliate_tag}} — Amazon affiliate tag (e.g. "freidoras-21")
{{current_year}}       — Year at build time (e.g. "2026")
```
