# S01: rsync + Caddy Deployment Service — UAT

**Milestone:** M004
**Written:** 2026-03-14

## UAT Type

- UAT mode: mixed (artifact-driven + live-runtime)
- Why this mode is sufficient: Contract verification (TypeScript builds, export checks, error message format) is fully automated. Live-runtime verification (rsync to real VPS2, Caddy serving via IP) requires human execution because CI has no VPS2 SSH path. Both modes together prove the slice.

## Preconditions

1. VPS2 exists and is reachable via Tailscale SSH from VPS1
2. Caddy is installed on VPS2 (`caddy --version` returns something)
3. VPS2 global Caddyfile contains `import sites/*` (typically in `/etc/caddy/Caddyfile`)
4. `/etc/caddy/sites/` directory exists on VPS2 (or will be created by sudo tee)
5. The user running the admin panel / BullMQ worker on VPS1 has `sudo systemctl reload caddy` on VPS2 via SSH (passwordless sudo for `caddy reload` configured or SSH key has sudoer access)
6. `SSH_AUTH_SOCK` is set in the shell running the test script (Tailscale SSH agent forwarding active)
7. A site has been generated by `GenerateSiteJob` — `.generated-sites/<slug>/dist/` exists and is non-empty

### 6. RsyncService live deploy to VPS2 (human — requires VPS2 reachable)

Create a minimal test script on VPS1:

```bash
cat > /tmp/test-deploy.mjs << 'EOF'
import { RsyncService } from '/home/daniel/monster/packages/deployment/dist/index.js';

const svc = new RsyncService();
await svc.deploy(
  'freidoras-de-aire',        // slug — must have .generated-sites/<slug>/dist/
  'vps2.internal',            // replace with actual Tailscale hostname
  'root',                     // replace with actual VPS2 user
  '/var/www/sites'            // replace with actual sites root

console.log('rsync complete');
EOF
node /tmp/test-deploy.mjs
```

1. Ensure `.generated-sites/<slug>/dist/` is non-empty (run `ls .generated-sites/<slug>/dist/` to confirm).
2. Run the script.
3. **Expected console output:**

   - `[RsyncService]` prefixed rsync progress lines (file transfer list)
   - Final: `rsync complete`
4. On VPS2, verify files arrived:
   ```bash
   ssh root@vps2.internal "ls /var/www/sites/<slug>/dist/"
   ```
5. **Expected:** `index.html` and site files present.

### 7. CaddyService live virtualhost + serve via VPS2 IP (human — requires VPS2 reachable)

```bash
cat > /tmp/test-caddy.mjs << 'EOF'
import { CaddyService } from '/home/daniel/monster/packages/deployment/dist/index.js';

const svc = new CaddyService();
await svc.writeVirtualhost(
  'test.example.com',         // domain — Caddy will serve this virtualhost
  'freidoras-de-aire',        // slug — maps to /var/www/sites/<slug>/dist
  'vps2.internal',            // replace with actual VPS2 Tailscale hostname
  'root'                      // replace with actual VPS2 user
);
console.log('Caddy virtualhost written and reloaded');
EOF
node /tmp/test-caddy.mjs
```

1. Ensure `SSH_AUTH_SOCK` is set (`echo $SSH_AUTH_SOCK` returns a path).
2. Run the script.
3. **Expected console output:**
   - `[CaddyService] connecting to vps2.internal as root`
   - `[CaddyService] writing /etc/caddy/sites/test.example.com.caddy`
   - `[CaddyService] reloading caddy on vps2.internal`
   - Final: `Caddy virtualhost written and reloaded`
4. On VPS2, verify the Caddyfile was written:
   ```bash
   ssh root@vps2.internal "cat /etc/caddy/sites/test.example.com.caddy"
   ```
   **Expected:** Contains `test.example.com {`, `root * /var/www/sites/freidoras-de-aire/dist`, `file_server`, `encode zstd gzip`.
5. Verify Caddy is serving:
   ```bash
   curl -H "Host: test.example.com" http://<vps2-public-ip>/
   ```
   **Expected:** Returns HTTP 200 with the site's HTML content. (Or use `curl http://<vps2-ip>/` if Caddy serves on the default virtualhost.)

## Edge Cases

### SSH_AUTH_SOCK not set (CaddyService)

1. Unset the agent: `SSH_AUTH_SOCK= node /tmp/test-caddy.mjs`
2. **Expected:** Script throws with a connection error from `node-ssh`. Error message should surface that SSH agent authentication failed. Does NOT hang indefinitely — `node-ssh` connect timeout applies.

### RsyncService with missing source directory

1. Run `svc.deploy('nonexistent-slug', ...)` where `.generated-sites/nonexistent-slug/dist/` does not exist.
2. **Expected:** rsync exits non-zero (code 23 or similar). Error message contains the exit code and rsync's "No such file or directory" in the stderr section. Thrown error is catchable — does not crash the process.

### CaddyService reload fails (bad Caddy config on VPS2)

1. Write a deliberately broken Caddyfile to `/etc/caddy/sites/bad.caddy` on VPS2.
2. Call `writeVirtualhost('bad2.example.com', ...)`.
3. **Expected:** `[CaddyService] reload step failed (exit N)` thrown error with Caddy's validation error text in the SSH output. The write step itself succeeds — the error is correctly identified as a reload failure, not a write failure.

## Failure Signals

- `RsyncService` or `CaddyService` not exported: `typeof m.RsyncService` returns `'undefined'`
- Build exits non-zero: any TypeScript error in `packages/deployment`, `packages/agents`, or `apps/admin`
- Error message missing exit code/host/stderr: diagnostic surface degraded — future debugging will be blind
- Settings VPS2 card absent from `/settings`: form schema and/or constants not extended correctly
- Supabase `settings` table missing `vps2_*` rows after save: upsert logic broken
- `curl http://<vps2-ip>/` returns connection refused or 404: Caddy not running, `import sites/*` missing, or files not in expected path

## Notes for Tester

- Tests 1–4 are automated and can be run directly from VPS1 (or any machine with the repo). Tests 5–7 require VPS2 to be reachable and Caddy to be installed.
- For test 7, using `Host: test.example.com` in the curl header lets you test against VPS2's IP without changing DNS — Caddy matches on the Host header.
- If `SSH_AUTH_SOCK` is not set in your terminal session (common in non-interactive pm2 sessions), start the SSH agent: `eval $(ssh-agent) && ssh-add`.
- The Caddyfile snippet written by `CaddyService` does NOT include the domain's TLS block — Cloudflare terminates TLS. Caddy serves plain HTTP on VPS2, Cloudflare proxies it with SSL. This is the intended architecture.
- If test 7 returns Caddy's default page instead of the site, `import sites/*` is missing from VPS2's global Caddyfile.
